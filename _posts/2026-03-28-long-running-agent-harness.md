Anthropic은 장기 실행 에이전트의 핵심 문제를 해결하는 두 가지 하네스 아키텍처를 공개했습니다. 2025년 11월 첫 번째 블루프린트를 발표한 후, 2026년 3월 더 발전된 GAN 기반 3에이전트 구조를 추가했습니다. 이 글은 두 아키텍처 모두를 다룹니다.

## 에이전트가 장기 작업에서 실패하는 이유

에이전트는 세션 단위로 동작합니다. 새로운 컨텍스트 창이 시작될 때 이전 작업에 대한 기억이 없습니다. 교대 근무자가 인수인계 문서 없이 작업을 이어받는 것과 같습니다.

Anthropic이 진단한 두 가지 핵심 실패 모드가 있습니다.

**컨텍스트 불안(Context Anxiety):** 컨텍스트 창이 채워질수록 에이전트는 작업을 조기에 마무리하려는 경향을 보입니다. 실제로 완료되지 않았음에도 완료됐다고 선언합니다. 이 문제는 Claude Sonnet 4.5에서 심각하여, Anthropic은 컴팩션(compaction) 대신 전체 컨텍스트 리셋을 선택했습니다. 컴팩션은 이전 대화를 요약하여 그 자리에서 압축하지만, 쌓인 컨텍스트의 무게는 여전히 남아 불안감이 지속됩니다. 리셋은 완전한 빈 슬레이트를 제공하지만, 견고한 인수인계 아티팩트가 필요합니다.

**자기 평가 실패(Self-Evaluation Failure):** 에이전트에게 자신의 작업을 평가하게 하면, 인간 관찰자에게는 명백히 평범한 수준임에도 자신 있게 칭찬합니다. 생성자가 자기 비판적이 되도록 튜닝하는 것은 매우 어렵습니다.

## Stage 1: 2에이전트 하네스 (2025년 11월)

첫 번째 해법은 두 개의 특화된 에이전트로 구성됩니다.

### Initializer Agent (최초 1회만 실행)

- `init.sh` 생성: 개발 환경 구동 스크립트
- `claude-progress.txt` 생성: 완료된 작업 문서화
- 200개 이상의 기능이 담긴 JSON 기능 목록 생성, 각 항목 초기 상태는 "failing"
- 초기 git 커밋으로 베이스라인 확립

### Coding Agent (이후 매 세션 실행)

- `init.sh`, git 로그, 진행 상황 파일을 읽어 현재 상태 파악
- 가장 높은 우선순위의 미완료 기능 하나 선택
- **세션당 기능 하나만** 구현
- 단위 테스트가 아닌 브라우저 자동화(Puppeteer MCP)로 엔드투엔드 테스트 실행
- 깔끔하고 병합 가능한 코드를 설명적인 커밋 메시지와 함께 커밋
- 진행 문서 업데이트

### 핵심 설계 결정들

기능 목록에 JSON 형식을 사용하는 이유가 있습니다. 모델은 마크다운보다 JSON을 불적절하게 수정할 가능성이 낮습니다. 여기에 에이전트가 테스트 항목을 삭제하거나 편집하지 못하도록 강한 지시를 추가했습니다.

매 세션의 시작 순서가 정해져 있습니다. `pwd` 실행, git 로그 읽기, 진행 파일 읽기, 기능 선택, `init.sh` 실행, 스모크 테스트. 이 순서가 에이전트가 현재 상태를 정확히 파악하게 해줍니다.

"클린 상태"의 정의도 중요합니다. 코드가 메인 브랜치에 병합 가능한 상태, 즉 주요 버그 없음, 잘 문서화됨, 기술 부채 없음을 의미합니다. 에이전트는 항상 이 상태를 유지하며 작업합니다.

## Stage 2: GAN 기반 3에이전트 하네스 (2026년 3월)

Anthropic 연구원 Prithvi Rajasekaran이 생성적 적대 신경망(GAN)에서 영감을 받아 3에이전트 아키텍처를 개발했습니다.

### Planner Agent

한 줄 프롬프트를 입력받아 상세한 제품 명세서로 확장합니다. 예를 들어 10개 스프린트에 걸친 16개 기능으로 구성된 스펙을 작성합니다.

### Generator Agent

React, FastAPI, PostgreSQL을 사용해 기능을 구현합니다. 각 스프린트 시작 전, Evaluator와 "스프린트 계약(sprint contract)"을 협상합니다. 이 계약은 테스트 가능한 성공 기준을 명확히 정의합니다.

### Evaluator Agent

핵심 역할입니다. Playwright MCP를 사용해 실행 중인 애플리케이션을 직접 탐색하고, 스크린샷을 찍고, 기능을 테스트합니다. 생성-평가 개선 루프를 스프린트당 5~15회 반복합니다.

평가는 4가지 기준으로 이루어집니다.

- 디자인 품질
- 독창성
- 완성도
- 기능 작동 여부

여기에 "AI 슬롭 패턴"에 대한 명시적 감점이 추가됩니다. 흰 카드 위 보라색 그라디언트, 수정 없이 그대로 사용한 스톡 컴포넌트 등이 해당합니다.

### 실제 성능 비교

| 방식 | 소요 시간 | 비용 | 결과 품질 |
|---|---|---|---|
| 단독 에이전트 (하네스 없음) | 20분 | $9 | 핵심 게임플레이 미작동 |
| 3에이전트 전체 하네스 | 6시간 | $200 | 완전 기능, 완성도 높은 UI |

Evaluator가 잡은 실제 버그 사례가 있습니다. "직사각형 채우기 도구가 드래그 구간 전체를 채우지 않고 시작/끝 지점에만 타일을 놓는" 문제였습니다. 코드 리뷰에서는 보이지 않는 버그입니다.

프론트엔드 디자인 실험에서 창의적인 돌파구도 있었습니다. 네덜란드 미술관 사이트를 10번 반복 개선하는 과정에서, Generator가 CSS 원근감으로 3D 방을 만들고 벽에 작품을 배치하고 문을 통해 이동하는 UI를 자체적으로 만들어냈습니다.

## 더 좋은 모델로의 진화

Claude Opus 4.6의 향상된 계획 능력과 장기 컨텍스트 신뢰성 덕분에, Anthropic은 스프린트 구조 전체를 제거하면서도 Planner와 Evaluator는 유지했습니다.

이 방식으로 구축한 DAW(디지털 오디오 워크스테이션) 애플리케이션은 3.8시간, $124.70의 비용으로 완성되었습니다.

여기서 중요한 원칙이 도출됩니다. **하네스의 복잡도는 모델 능력이 향상될수록 감소해야 합니다.**

---

## 핵심 원칙 7가지

**1. 하네스 설계가 이제 경쟁 우위입니다**
동일한 모델도 하네스 품질에 따라 결과가 극적으로 달라집니다. 모델을 선택하는 것만큼 하네스를 설계하는 능력이 중요해졌습니다.

**2. 컨텍스트 리셋 vs 컴팩션**
리셋은 빈 슬레이트를 제공하지만 견고한 인수인계 아티팩트가 필요합니다. 컴팩션은 연속성을 유지하지만 컨텍스트 불안이 지속됩니다. 작업 성격에 따라 선택이 달라집니다.

**3. 평가자와 생성자를 분리하세요**
독립적인 회의적 평가자를 튜닝하는 것이 생성자를 자기 비판적으로 만드는 것보다 훨씬 쉽습니다. GAN 구조가 이 통찰을 그대로 반영합니다.

**4. 아티팩트가 인수인계 메커니즘입니다**
git 커밋, 진행 파일, 기능 목록이 에이전트가 세션을 넘어 작업을 재개할 수 있게 해줍니다. 이 파일들의 품질이 시스템 전체 신뢰성을 결정합니다.

**5. 사용자처럼 테스트하세요**
브라우저 자동화(Playwright, Puppeteer)는 코드 리뷰가 놓치는 버그를 잡습니다. 단위 테스트만으로는 충분하지 않습니다.

**6. 한 번에 하나씩, 항상 병합 가능한 상태로**
세션당 기능 하나 구현, 항상 메인에 병합 가능한 상태 유지. 한 번에 전부 구현하려는 시도보다 점진적 접근이 더 안정적입니다.

**7. 미해결 질문**
범용 단일 에이전트와 특화된 멀티에이전트 파이프라인(테스트 에이전트, QA 에이전트, 정리 에이전트) 중 어느 쪽이 더 나은지는 아직 결론이 나지 않았습니다. Anthropic은 이를 열린 질문으로 남겨두었습니다.
