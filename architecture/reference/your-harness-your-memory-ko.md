# Your harness, your memory (한글 번역)

- 원문 파일: `/workspace/group/uploads/1776094599567/Your_harness_your_memory.md`
- 원문 작성자: Harrison Chase (@hwchase17)
- 번역 목적: 에이전트 하네스와 메모리 소유권 논지를 한국어로 공유

## 제목
당신의 하네스, 당신의 메모리

## 요약
에이전트를 만들 때 하네스(agent harness)는 점점 표준이 되고 있으며 사라지지 않는다. 하네스는 메모리와 강하게 결합되어 있다. 특히 폐쇄형 하네스나 공급자 API 뒤에 숨겨진 하네스를 쓰면, 에이전트 메모리 통제권을 제3자에게 넘기게 된다. 메모리는 에이전트 경험의 품질과 고착성(stickiness)을 좌우하므로 강한 락인을 만든다. 따라서 메모리, 그리고 메모리를 다루는 하네스는 개방적이어야 하며 개발자가 직접 소유해야 한다.

## 1) 에이전트 하네스는 에이전트 구축의 중심이며 앞으로도 유지된다
지난 3년간 에이전트 시스템의 "최선의 방법"은 계속 변했다.

- 초기: 단순 RAG 체인 중심
- 이후: 더 복잡한 플로우(그래프 기반 오케스트레이션)
- 최근: 모델 성능 향상으로 하네스 중심 구조가 부상

글에서 예시로 든 하네스 생태계:
- Claude Code
- Deep Agents
- Pi / OpenClaw
- OpenCode
- Codex
- Letta Code

핵심 주장:
- 모델이 좋아져도 하네스는 사라지지 않는다.
- 2023년에 필요했던 일부 스캐폴딩은 줄었지만, 그 자리를 다른 종류의 스캐폴딩이 대체한다.
- 에이전트는 본질적으로 "LLM + 도구/데이터 상호작용"이므로 이를 중재하는 시스템(하네스)은 항상 필요하다.
- 모델 API에 내장된 웹검색 같은 기능도 "모델 그 자체"가 아니라 API 뒤의 경량 하네스가 tool calling으로 조정하는 성격이다.

## 2) 하네스와 메모리는 분리된 플러그인이 아니라 본질적으로 결합된다
글은 "memory isn’t a plugin (it’s the harness)" 관점을 지지한다.

핵심 논지:
- 메모리는 컨텍스트 관리의 한 형태이며, 컨텍스트 관리는 하네스 핵심 책임이다.
- "메모리를 하네스에 끼워 넣는다"는 질문은 자동차에 "운전"을 플러그인으로 넣겠다는 말에 가깝다.

하네스가 실제로 메모리에 관여하는 지점 예시:
- AGENTS.md / CLAUDE.md 같은 지침 파일을 어떤 방식으로 컨텍스트에 로딩하는가
- 스킬 메타데이터를 어디에 어떻게 노출하는가
- 에이전트가 시스템 지침을 수정 가능한가
- 압축(compaction) 시 무엇이 남고 사라지는가
- 상호작용 로그가 저장/검색 가능한가
- 메모리 메타데이터가 에이전트에 어떤 형태로 보이는가
- 현재 작업 디렉터리와 파일시스템 정보를 얼마나 노출하는가

또한 장기 메모리는 아직 산업적으로 초기 단계라 공통 추상화가 충분히 성숙하지 않았고, 당분간은 하네스 중심으로 메모리를 다루는 모델이 현실적이라고 본다.

## 3) 하네스를 소유하지 못하면 메모리도 소유하지 못한다
강조점:
- 폐쇄형 하네스(특히 API 뒤의 상태 저장형 구조)는 메모리 주권을 약화시킨다.

문제 단계:
1. 상대적으로 약한 문제
- 공급자 서버에 상태를 저장하는 API를 쓰면, 모델 교체 후 동일 스레드 연속성이 깨질 수 있다.

2. 더 큰 문제
- 폐쇄형 하네스는 내부 메모리 처리 방식이 불투명해서 다른 하네스로 이식하기 어렵다.

3. 최악의 경우
- 하네스와 장기 메모리 전체가 API 뒤에 있으면 메모리 가시성/제어권을 거의 잃는다.
- 어떤 데이터가 어떻게 저장·요약·사용되는지 통제하기 어렵고, 플랫폼 락인이 강해진다.

저자 관점:
- 모델 공급자는 메모리를 통한 락인 유인이 크며, 실제로 점점 더 많은 메모리/하네스 기능을 API 뒤로 이동시키고 있다.

## 4) 메모리는 차별화와 락인의 핵심 자산이다
메모리는 단순 기능이 아니라 제품 경쟁력의 핵심으로 제시된다.

- 사용자 상호작용이 누적되며 에이전트가 개인화된다.
- 경험 품질이 시간이 갈수록 향상된다.
- 동일한 모델/도구를 쓰는 경쟁자와의 차별화 데이터 자산이 된다.

핵심 문장 요지:
- 메모리가 없으면 에이전트는 쉽게 복제 가능하다.
- 메모리가 있으면 고유 데이터 플라이휠이 형성된다.

사례:
- 내부 이메일 에이전트가 삭제되자 동일 템플릿으로 재생성해도 사용자 취향/톤 학습이 사라져 품질이 급격히 하락했다.
- 이를 통해 메모리의 "고착성"과 제품 가치가 확인됐다는 내용.

## 5) 결론: Open Memory, Open Harnesses
권고:
- 메모리는 에이전트 경험을 만드는 개발자가 직접 소유해야 한다.
- 하네스/메모리는 특정 모델 공급자와 분리해 모델 선택권(옵셔널리티)을 유지해야 한다.

글에서 제시한 Deep Agents 방향:
- 오픈소스
- 모델 불가지론(model-agnostic)
- open standards(agents.md, skills) 지향
- Mongo/Postgres/Redis 등 다양한 메모리 스토어 연동
- 배포 유연성(자체 호스팅, 클라우드 선택, BYO DB)

최종 메시지:
- "메모리를 소유하려면 오픈 하네스를 써야 한다."

## 아키텍처 관점에서의 실무 함의
현재 `notes/architecture` 기준으로 연결하면 다음 원칙이 강화된다.

1. 하네스-상태 책임 분리 유지
- 하네스는 무상태 이벤트 발행 중심, 상태 확정은 State plane 단일 권한.

2. 메모리 데이터 소유권 보장
- 세션/이벤트/승인 근거는 append-only 및 이식 가능한 저장 구조 유지.

3. 공급자 종속 최소화
- 특정 벤더의 서버측 메모리 기능에 종속되지 않도록 인터페이스/스키마를 내부 기준으로 고정.

4. 자동화 범위 제한 + 사용자 통제 우선
- 고위험 조작은 승인/grant를 거치고, 자동 복구는 제한 범위에서만 수행.
